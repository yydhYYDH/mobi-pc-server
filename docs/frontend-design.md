# Frontend Design

本文档描述当前前端控制台的产品设计、交互逻辑和后续规范化方向。目标是让前端成为一个清晰、可靠的本地开发控制台，而不是把后端进程、模型文件和设备连接细节直接暴露给用户。

## 1. 产品定位

你的智伴前端是本地控制台，服务对象是需要在 PC 上管理本地推理服务、ModelScope 模型和 HarmonyOS 设备连接的开发者。

前端的核心职责：

- 展示本地系统状态：推理服务、当前模型、模型下载状态、HDC 设备、运行日志。
- 发起明确动作：启动/停止服务、下载/删除/加载模型、连接/断开设备、发送对话测试请求。
- 降低排障成本：把错误、最近风险、日志过滤和运行路径放在容易发现的位置。

前端不应该承担的职责：

- 不拼接本地 shell 命令。
- 不直接读写模型目录或日志文件。
- 不维护一套脱离后端的长期业务状态。
- 不提供任意命令执行入口。

## 2. 当前信息架构

当前应用采用左侧固定导航 + 右侧工作区的控制台布局，包含 7 个视图：

| 视图 | 当前用途 | 设计定位 |
| --- | --- | --- |
| 总览 | 展示整体状态、快捷启动、模型选择、最近风险 | 默认首页，回答“现在能不能跑起来” |
| 模型 | 展示 ModelScope 目录、本地下载状态、下载/加载/删除动作 | 资产管理页 |
| 推理服务 | 展示服务状态、后端、端口、托管方式、当前模型、启动/停止 | 运行时控制页 |
| 设备 | 展示 HDC 可用性、设备列表、自动搜索、连接/断开、LLM 端口 | HarmonyOS 连接页 |
| 对话测试 | 调用本地 OpenAI-compatible endpoint，验证模型可用性 | 最小推理验证页 |
| 日志 | 展示后端、推理服务、ModelScope、HDC 日志，支持过滤、自动滚动、复制 | 排障页 |
| 设置 | 展示 API、端口、路径、平台和 LLM 端口 | 运行环境页 |

建议保持这个信息架构。它符合“先看状态、再处理模型/服务/设备、最后验证和排障”的开发者工作流。

## 3. 关键用户路径

### 3.1 首次启动并跑通模型

1. 打开应用，进入“总览”。
2. 总览展示推理服务状态、HDC 状态、本地模型数量和最近错误。
3. 如果没有可启动模型，用户进入“模型”页下载模型。
4. 模型下载完成后，回到“总览”选择模型。
5. 点击“加载并启动”，后端负责启动所选推理后端并加载模型。
6. 进入“对话测试”发送一条消息，验证 OpenAI-compatible endpoint 是否可用。
7. 如失败，进入“日志”页过滤错误。

### 3.2 已有模型的日常使用

1. 总览选择本地已下载模型。
2. 根据当前服务状态展示主按钮：
   - 服务未运行：加载并启动。
   - 服务运行但模型不同：切换并重启。
   - 当前模型已运行：当前模型运行中，按钮禁用。
3. 用户可以从总览快速进入日志或设备页处理异常。

### 3.3 HarmonyOS 设备连接

1. 应用首次获得 HDC 状态后会尝试自动搜索设备。
2. 设备页展示 HDC 是否可用、设备数量、手机 LLM URL、端口转发状态。
3. 用户可输入设备序列号或 `host:port` 手动连接。
4. 用户可设置 LLM server port，该配置写入 `localStorage`，并被设备连接动作复用。

## 4. 状态模型

前端状态应分为三类。

### 4.1 后端事实状态

这些状态必须以后端 API 为准：

- `runtimeStatus`: 当前推理服务状态、端口、后端、当前模型、托管方式、消息。
- `models`: 模型目录。
- `localModels`: 本地模型可用性。
- `downloads`: 下载队列和进度。
- `hdc`: HDC 可用性、设备列表、LLM 端口转发状态。
- `logs`: 日志内容。

当前实现通过 `load()` 并发读取这些 API。后续可以迁移到 TanStack Query，但仍应保持“后端是事实来源”。

### 4.2 UI 临时状态

这些状态只影响当前交互：

- 当前视图 `activeView`。
- 操作忙碌状态 `serverBusy`、`modelBusy`、`deviceBusy`、`chatBusy`。
- 表单输入 `hdcTarget`、`hdcLlmPort`、`chatInput`。
- 提示与错误 `error`、`deviceNotice`、`chatError`。
- 日志过滤和自动滚动。

这类状态不应被后端持久化，除非它代表真实配置。目前 `hdcLlmPort` 使用 `localStorage` 是合理的，因为它是用户偏好。

### 4.3 派生状态

派生状态不要单独存储，应该从后端事实状态计算：

- `downloadedCount`
- `launchableModels`
- `activeModelName`
- `connectedDevices`
- `systemReady`
- `criticalLog`
- `visibleLogLines`

这可以减少状态不同步问题。

## 5. 交互规则

### 5.1 全局加载与刷新

- 页面首次加载时请求所有核心状态。
- 只在下载活跃或日志页打开时轮询：
  - 下载中：约 1.5 秒刷新。
  - 日志页：约 3 秒刷新。
- 顶部“刷新”按钮应始终可用，失败时展示全局错误。
- 后续建议加入 WebSocket 或 SSE，把下载进度和日志更新改为事件驱动。

### 5.2 操作互斥

当前通过 `serverBusy`、`modelBusy`、`deviceBusy` 避免重复点击。规范如下：

- 启动/停止服务期间，禁用会改变推理进程状态的操作。
- 模型下载、加载、删除期间，禁用同一模型相关动作；必要时禁用全部模型动作，避免并发破坏本地文件。
- 设备自动搜索、连接、断开期间，禁用其他设备动作。
- 对话生成期间，禁用发送按钮，但允许编辑下一条输入可以作为后续优化。

### 5.3 按钮文案

按钮文案必须反映当前动作，而不是只反映目标状态：

- `启动服务` -> `启动中...` -> `服务运行中`
- `停止服务` -> `停止中...`
- `加载并启动` / `切换并重启` / `当前模型运行中`
- `下载` -> `处理中...`
- `自动搜索` -> `搜索中...`
- `发送` -> `生成中`

### 5.4 错误呈现

错误应分层展示：

- 全局 API 加载失败：工作区顶部红色提示。
- 设备连接失败或提示：设备面板内的 `deviceNotice`。
- 对话请求失败：对话面板内的 `chatError`。
- 后端或推理服务运行错误：总览“最近风险”和日志页。

后续建议把 HTTP 错误中的 `detail` 解析出来，不只展示 `HTTP status`。

## 6. 页面设计规范

### 6.1 总览

总览是默认工作台，应该优先展示可行动信息：

- 顶部工作区：当前模型或“尚未加载模型”、推理服务/HDC/设备摘要、启动/停止/管理模型动作。
- 模型启动区：选择本地可启动模型，展示模型来源和大小。
- 指标卡：推理服务、当前模型、本地模型、HarmonyOS 设备。
- 快捷操作：下载模型、自动搜索设备、查看 HDC、打开日志。
- 最近风险：从日志中提取最近 error/failed/exception/timeout。

总览不应成为所有功能的重复入口，只保留最常用、最能恢复运行状态的动作。

### 6.2 模型页

模型页以表格为主，适合扫描和批量管理：

- 模型列：名称、ModelScope ID、描述或状态消息。
- 状态列：未下载、排队中、下载中、校验中、已下载、失败。
- 进度列：字节进度、百分比、进度条。
- 操作列：下载、加载、删除。

后续建议：

- 增加运行时标签，例如 `llama.cpp CUDA`、`MobiInfer`。
- 对删除动作增加确认，避免误删大模型。
- 下载失败时提供重试，不要只显示“下载”。
- 当模型正在运行时，删除按钮必须禁用并说明原因。

### 6.3 推理服务页

服务页用于确认运行时细节：

- 状态徽标放在标题右侧。
- `dl` 展示端口、托管方式、当前模型、消息。
- 操作只保留启动和停止。

后续可以增加：

- 推理后端二进制路径。
- 启动参数摘要。
- 最近启动耗时。
- 当前服务是否由后端托管，若是外部进程，停止按钮需要谨慎处理。

### 6.4 设备页

设备页应围绕“能否把手机连到本地 LLM 服务”设计：

- HDC 是否存在。
- 当前设备列表。
- 手机 LLM URL。
- LLM 端口转发是否已建立。
- 自动搜索为主操作，手动连接为补充。

设备输入规范：

- 设备目标：序列号或 `host:port`。
- LLM 端口：数字，范围 1-65535，默认 8088。
- 输入校验失败时在面板内提示，不触发 API。

### 6.5 对话测试页

对话测试页是健康检查工具，不是完整聊天产品：

- 只用于验证当前推理服务和模型是否响应。
- 服务未运行或无端口时，发送前阻断并提示。
- 流式返回时逐 token 追加到最后一条 assistant 消息。

后续建议：

- 显示当前 active model。
- 支持停止生成。
- 支持清空会话。
- 请求应优先通过后端代理，避免前端直接请求推理服务端口导致跨域或环境差异。

### 6.6 日志页

日志页用于排障：

- 默认展示最近日志。
- 支持关键词过滤。
- 支持自动滚动。
- 支持复制当前过滤结果。

后续建议：

- 区分 backend、llm、modelscope、hdc 日志源。
- 支持 severity 过滤。
- 复制成功后给出轻量反馈。

### 6.7 设置页

设置页当前以只读运维信息为主：

- 前端 API base。
- 后端/推理服务端口。
- 推理服务托管来源。
- HDC 路径。
- LLM server 端口。
- 手机访问地址。
- 桌面平台。

后续需要明确哪些配置可编辑，哪些只是状态展示。可编辑项应保存到后端配置或用户本地偏好，不要混在只读信息里。

## 7. 视觉设计规范

当前视觉方向是“开发者控制台”：浅灰背景、白色面板、蓝色主按钮、绿色成功态、红色错误态。建议继续保持克制、密集、可扫描。

### 7.1 布局

- 桌面端：左侧导航 260px，右侧工作区自适应。
- 中等屏幕：导航横向宫格，内容单列。
- 移动端：所有面板单列，按钮可换行或占满宽度。
- 面板圆角不超过 8px，避免营销页式大卡片。

### 7.2 颜色语义

- 主操作：蓝色 `#245a9d`。
- 成功/运行：绿色。
- 进行中：蓝色浅底。
- 停止/空闲：灰色。
- 错误/失败：红色。
- 背景：浅灰，面板：白色。

颜色必须服务状态语义，避免引入无语义装饰色。

### 7.3 文案

- 页面标题使用中文，例如“总览”“模型”“推理服务”。
- 小型技术标签可以保留英文，例如 `ModelScope`、`Runtime`、`OpenAI-compatible endpoint`。
- 错误文案要说明动作和结果，例如“启动失败：HTTP 500”。
- 后续应尽量展示后端返回的具体错误原因。

## 8. 组件化建议

当前页面已经拆出了多个函数组件，但所有类型、请求、状态和视图都在 `frontend/src/main.tsx`。建议按下面结构逐步拆分：

```text
frontend/src/
  app/
    App.tsx
    navigation.ts
  api/
    client.ts
    models.ts
    runtime.ts
    devices.ts
    logs.ts
    types.ts
  components/
    Alert.tsx
    Button.tsx
    MetricCard.tsx
    Panel.tsx
    StatusPill.tsx
    ProgressBar.tsx
  views/
    OverviewView.tsx
    ModelsView.tsx
    ServerView.tsx
    DevicesView.tsx
    ChatView.tsx
    LogsView.tsx
    SettingsView.tsx
  styles/
    base.css
    layout.css
    components.css
```

拆分优先级：

1. 先拆 API client 和类型，避免每个视图直接写 `fetch`。
2. 再拆状态徽标、面板、按钮、进度条等重复 UI。
3. 最后拆各视图，保持视图只描述布局和事件回调。

## 9. API 设计对前端的要求

前端需要稳定结构化响应。建议后端继续保证：

- 所有状态字段使用枚举或稳定字符串。
- 所有动作接口返回动作后的最新状态，减少额外刷新。
- 错误响应包含可展示的 `detail`。
- 下载、日志、设备搜索这类长任务提供事件流。

前端应该封装 API 错误解析：

```ts
async function parseApiError(response: Response, fallback: string) {
  try {
    const body = await response.json();
    return body?.detail ?? fallback;
  } catch {
    return fallback;
  }
}
```

## 10. 近期改进清单

- 将 `main.tsx` 拆分为 API、组件、视图三层。
- 增加 `StatusPill`、`Panel`、`MetricCard`、`ProgressBar` 基础组件。
- 解析并展示后端 `detail` 错误。
- 删除模型前增加确认。
- 对话测试增加清空和停止生成。
- 日志复制增加成功/失败反馈。
- 使用 SSE 或 WebSocket 替代下载和日志轮询。
- 将前端直接访问推理服务 chat endpoint 改为后端代理。
- 引入 TanStack Query 时，统一缓存 key 和刷新策略。
